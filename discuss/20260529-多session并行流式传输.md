- "可以对claude也生效"
- "归档提交"

  - Claude `/clear` 不发送给 Claude CLI，而是在当前 Web 会话下生成新的 `cliSessionId`，设置 `cliSessionReady=false`，下一条普通消息会用 `--session-id` 创建新的 Claude CLI session。

- `/clear` 只切换底层 CLI session/thread，不清空 Web 侧可见聊天历史，满足“原会话下新开 session”。

## 2026-06-13 12:52 - 上下文用量展示修正、Status 明细与子消息计时

### 用户需求
- "上下文显示变成225%了"
- "255%是否正常"
- "问题根因是啥"
- "status 命令里面加上上下文使用的详细信息"
- "聊天框里面的assistant的每个子消息能在标题栏显示下开始时间和最终耗时吗"
- "归档提交"

### 问题与根因
- Codex rollout JSONL 的 `total_token_usage.total_tokens` 是线程累计 token，不是当前上下文窗口占用。
- 旧逻辑曾把累计 token 或异常 token_count ratio 直接除以 `model_context_window` 展示，因此会出现 225%、255% 这类不可能的上下文占用。
- 修正提交之后，运行中的 `node server.js` 进程如果未重启，仍会继续用旧逻辑落盘异常 `contextUsage`。
- 前端原先不校验 `contextUsage.percent`，历史消息里已经保存的异常百分比会被继续展示。

### 实现内容
- `server.js`
  - 新增 `isValidContextUsage()`，只允许 0%-100% 的上下文 ratio 用于展示、归档阈值判断和消息 `contextUsage`。
  - `/status` 输出新增 `Context usage` 明细：当前窗口 token、窗口大小、当前窗口使用率、原始 ratio、归档阈值、线程累计 token 和最近 token_count 事件时间。
  - 原始 ratio 超出 0%-100% 时在 `/status` 标记为 ignored，避免把异常口径当作真实上下文占用。
  - 为 assistant 子消息新增计时字段：`thinkingMeta.startedAt/durationMs`，以及 tool 的 `startedAt/endedAt/durationMs`。
  - Codex subagent、Codex command/tool、Claude thinking/tool 均保存开始时间；有结束事件时记录结束时间，assistant 结束时补齐未完成 tool 的最终耗时。
- `public/app.js`
  - 前端只展示形状完整且 ratio 在 0%-100% 范围内的上下文百分比，隐藏历史异常值。
  - assistant 气泡内的 thinking 和 tool 子块标题栏改为标题 + 元信息布局。
  - 子块标题栏展示开始时间；收到结果或最终消息后展示耗时。
  - 历史消息缺少子项时间字段时不显示假时间。
- `public/style.css`
  - 为 thinking/tool 标题栏增加左右布局、标题省略和小号弱化时间文本样式。
- `public/index.html`
  - 资源版本更新为 `style.css?v=88`、`app.js?v=96`。

### 关键设计判断
- 超过 100% 的 usage 不再 clamp 成 100%，而是直接视为不可用；这样不会掩盖上游 token_count 字段口径异常。
- `/status` 同时展示有效当前窗口占用和原始 ratio，便于后续排查 Codex CLI 事件口径变化。
- 子消息时间只基于 RemoteCodex 观察到的流式事件时间，不声称是 Codex/Claude 内部真实执行起止时间。
- 现有历史消息不会补写时间字段，避免制造不准确历史数据。

### 问题
- Web 端原先只拦截并支持 Codex `/status`，用户输入 `/effort` 会返回不支持。
- Codex `exec` 非交互模式没有独立的 `--effort` 参数，但支持通过 `-c model_reasoning_effort="..."` 覆盖配置。
- 初版 `/effort` 支持只允许 `minimal/low/medium/high`，遗漏了本机 Codex CLI 实测可用的 `xhigh`。

## 2026-05-30 00:00 - 多 Session 并行流式传输重新设计实现（已提交）

### 设计原则
- 避免重复上次一次性大改后难定位的问题
- 服务端先保证流式进程按 session 隔离
- 每个流式事件都携带 `sessionId`
- 前端按 `sessionId` 分流，非当前 session 的流式事件只更新缓存，不写当前 DOM
- 当前 session 切换回来时，根据缓存重建流式骨架
- 不改 session 文件结构，不改 REST API 的既有响应结构

### 服务端实现
- `activeProcesses` 从 socket 维度改为 `${socket.id}:${sessionId}` 维度
- 新增 `socketProcesses`，用于 disconnect 时批量清理该 socket 下所有 session 的 CLI 进程
- `load_session` 返回原 session 数据并附加 `isStreaming`
- `send_message` 每次按 payload 的 `sessionId` 重新加载目标 session，形成 `streamSession` 闭包
- CLI stdout/stderr/close/error 回调全部写入 `streamSession`，避免用户切换 session 后串写到新的 `currentSession`
- `message_added` 事件改为 `{ sessionId, message }`
- `stream_thinking_start`、`stream_thinking`、`stream_text`、`stream_tool_use`、`stream_tool_result`、`stream_end`、`stream_error` 均携带 `sessionId`
- `interrupt` 接收 `{ sessionId }`，只中断当前 session 的进程

### 前端实现
- 移除全局单一 `isStreaming`
- 新增 `streamingSessions: Map<sessionId, state>`
- 每个 streaming state 保存 assistant 占位消息、thinking、text、toolUses 和 thinking 状态
- 当前 session 的事件更新 DOM；非当前 session 的事件只更新 `streamingSessions`
- `renderMessages()` 在当前 session 仍在流式时跳过最后一个空 assistant 占位消息，并用缓存重建流式骨架
- 切换 session 时根据目标 session 的 streaming state 更新 Send/Stop 按钮
- 发送消息时先为当前 session 建立 streaming state，再发送 `{ sessionId, content }`
- Stop 按钮发送 `{ sessionId }` 精确中断
- 资源版本更新到 `app.js?v=73`、`style.css?v=77`
