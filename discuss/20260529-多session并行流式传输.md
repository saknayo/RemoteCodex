## 2026-05-31 08:21 - iOS Chrome Wake Lock Fallback（待提交）

### 用户需求
- "wake lock没生效，sending状态中还是熄屏了 ios google浏览器"
- "提交"

### 问题
- iOS Chrome 并不是普通 Chromium 内核，受 iOS WebKit 和系统节电策略限制
- Screen Wake Lock API 需要安全上下文，公网 HTTP 访问下原生 `navigator.wakeLock` 不可靠或不可用
- 当前实现只调用原生 Wake Lock，失败后只打印 warning，iOS Chrome 上没有兜底路径

### 实现方案
- 保留原生 `navigator.wakeLock.request('screen')` 作为优先方案
- 当 `navigator.wakeLock` 不存在或请求失败时，启用 fallback
- fallback 使用隐藏的 muted inline video：
  - 创建 2x2 canvas
  - 每秒轻微绘制变化
  - 使用 `canvas.captureStream(1)` 生成低频视频流
  - 将视频设置为 muted、playsInline，并以 1px 低透明度挂到页面
- 流式期间播放 fallback 视频，所有流式结束后停止视频、停止 stream track 并清理 interval
- 原生 Wake Lock 释放时同步释放 fallback
- 更新 `public/index.html` 中 `app.js` 版本到 v78

### 风险说明
- iOS Chrome 上网页仍无法 100% 保证不熄屏，fallback 也可能被系统策略限制
- 最可靠方案仍然是 HTTPS 安全上下文，或由用户在 iOS 设置里临时调长自动锁定时间

## 2026-05-30 00:00 - 多 Session 并行流式传输重新设计实现（已提交）

### 用户需求
- "恢复"
- "你来重新设计开发下多session并行功能"
- "继续"
- "状态如何"
- "我已经重启了服务端，上面就是重启后的现象"
- "我重启了"
- "提交"

### 设计原则
- 避免重复上次一次性大改后难定位的问题
- 服务端先保证流式进程按 session 隔离
- 每个流式事件都携带 `sessionId`
- 前端按 `sessionId` 分流，非当前 session 的流式事件只更新缓存，不写当前 DOM
- 当前 session 切换回来时，根据缓存重建流式骨架
- 不改 session 文件结构，不改 REST API 的既有响应结构

### 服务端实现
- `activeProcesses` 从 socket 维度改为 `${socket.id}:${sessionId}` 维度
- 新增 `socketProcesses`，用于 disconnect 时批量清理该 socket 下所有 session 的 CLI 进程
- `load_session` 返回原 session 数据并附加 `isStreaming`
- `send_message` 每次按 payload 的 `sessionId` 重新加载目标 session，形成 `streamSession` 闭包
- CLI stdout/stderr/close/error 回调全部写入 `streamSession`，避免用户切换 session 后串写到新的 `currentSession`
- `message_added` 事件改为 `{ sessionId, message }`
- `stream_thinking_start`、`stream_thinking`、`stream_text`、`stream_tool_use`、`stream_tool_result`、`stream_end`、`stream_error` 均携带 `sessionId`
- `interrupt` 接收 `{ sessionId }`，只中断当前 session 的进程

### 前端实现
- 移除全局单一 `isStreaming`
- 新增 `streamingSessions: Map<sessionId, state>`
- 每个 streaming state 保存 assistant 占位消息、thinking、text、toolUses 和 thinking 状态
- 当前 session 的事件更新 DOM；非当前 session 的事件只更新 `streamingSessions`
- `renderMessages()` 在当前 session 仍在流式时跳过最后一个空 assistant 占位消息，并用缓存重建流式骨架
- 切换 session 时根据目标 session 的 streaming state 更新 Send/Stop 按钮
- 发送消息时先为当前 session 建立 streaming state，再发送 `{ sessionId, content }`
- Stop 按钮发送 `{ sessionId }` 精确中断
- 资源版本更新到 `app.js?v=73`、`style.css?v=77`

### 验证
- `node --check server.js` 通过
- `node --check public/app.js` 通过
- `git diff --check` 通过
- 临时端口启动新服务，HTTP 首页返回 200
- 临时服务登录和 `/api/sessions` 列表接口验证通过
- 临时测试服务已清理
- 正式服务已重启到新后端进程
- 页面资源已更新到 `/app.js?v=73` 与 `/style.css?v=77`
- 重启后观察到同一后端下同时存在 Codex 和 Claude CLI 进程，说明并发启动生效
- Claude 进程结束后可正常落盘到对应 session 文件

### 已发现的后续风险
- 浏览器如果没有强刷，可能继续运行旧 JS 内存状态，导致前端选中 session 与发送的 `sessionId` 不一致
- 一次人工观察中，用户以为回复来自 `rc-cc`，但落盘结果显示实际写入 `ren-cc`；需后续用浏览器强刷后的干净状态再验证 session 选择一致性

### 待验证

## 2026-05-29 19:30 - 多 Session 并行流式传输实现尝试（已回退）

### 用户需求
- 实现多 Session 并行流式传输：一个 Session 正在流式输出时，切换到另一个 Session，后台流式不受影响；切回来时能恢复流式骨架继续显示

### 实现方案

#### server.js 改动
- `activeProcesses` key 从 `socket.id` 改为 `${socketId}:${sessionId}`，支持同一 socket 并行运行多个 CLI 进程
- 新增 `socketSessions` Map，按 socket.id 索引其所有活跃进程 key，便于 disconnect 时批量清理
- 流式回调使用 `streamSession` 闭包引用捕获 session 快照，避免 `load_session` 切换后流式数据写入错误 session
- 所有 `stream_*` emit 增加 `sessionId` 字段，客户端按 session 过滤
- `interrupt` 改为接收 `{ sessionId }` 精确中断
- `session_loaded` 返回 `isStreaming` 标记告知客户端该 session 是否有后台活跃流式

#### public/app.js 改动
- 新增 `streamingSessionIds` Set，用 sessionId 集合跟踪哪些 session 正在流式传输
- `resetStreamingState(targetSessionId)` 改为按 sessionId 清理
- 新增 `updateSendButton()`、`saveCurrentStreamingState()`、`restoreStreamingState()` 函数
- 所有 `stream_*` 事件增加 `sessionId` 过滤，非当前 session 的流式事件静默忽略
- `session_loaded`/`session_created` 处理器增加流式状态保存/恢复逻辑
